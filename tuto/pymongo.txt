source venv/bin/activate

🍃 3. Installer MongoDB
Option simple (locale)

sudo apt install mongodb -y

Puis démarre :

sudo systemctl start mongodb
sudo systemctl enable mongodb

Teste :

mongo

🔗 4. Connecter Python à MongoDB

Installe le driver :

pip install pymongo

from pymongo import MongoClient

Exactement ✅ !

En MongoDB, une collection est créée automatiquement la première fois que tu y insères un document. Tu n’as pas besoin de la créer manuellement comme en SQL.
Exemple concret

Dans mongosh :

use test_db       // sélectionne ou crée la base test_db
db.users.insertOne({nom: "Alice"})  // "users" n’existait pas encore

➡️ MongoDB va automatiquement :

    Créer la base test_db si elle n’existe pas

    Créer la collection users si elle n’existe pas

    Insérer le document {nom: "Alice"}

En Python avec pymongo

Même principe :

from pymongo import MongoClient

client = MongoClient("mongodb://localhost:27017/")
db = client["test_db"]       # base
collection = db["users"]     # collection (créée si n’existe pas)

collection.insert_one({"nom": "Bob"})  # insertion

➡️ La collection users sera créée automatiquement si elle n’existait pas déjà.

💡 Astuce : Une collection n’existe réellement qu’après l’insertion du premier document.
Si tu fais juste db.users sans insérer, elle n’apparaîtra pas quand tu feras show collections.

L’erreur mongod.service: Main process exited, status=14 signifie que MongoDB ne démarre pas. C’est souvent dû à problèmes de configuration ou de permissions du dossier de données (dbpath) sur Ubuntu.

On va résoudre ça étape par étape.
1️⃣ Vérifier le fichier de log

MongoDB écrit ses erreurs dans le fichier de log (très utile pour comprendre) :

cat /var/log/mongodb/mongod.log | tail -n 50

💡 Cherche des messages comme permission denied ou data directory not found.
2️⃣ Vérifier le dossier de données

Par défaut, MongoDB utilise /var/lib/mongodb comme dbPath.

Vérifie qu’il existe et que les permissions sont correctes :

ls -ld /var/lib/mongodb

Si tu vois que ce n’est pas accessible par mongodb, corrige :

sudo chown -R mongodb:mongodb /var/lib/mongodb
sudo chmod 700 /var/lib/mongodb

3️⃣ Vérifier le fichier de configuration

MongoDB lit /etc/mongod.conf

Vérifie le chemin de dbPath :

storage:
  dbPath: /var/lib/mongodb

Et les logs :

systemLog:
  destination: file
  path: /var/log/mongodb/mongod.log
  logAppend: true

Si ces dossiers n’existent pas → crée-les avec les bonnes permissions :

sudo mkdir -p /var/lib/mongodb
sudo mkdir -p /var/log/mongodb
sudo chown -R mongodb:mongodb /var/lib/mongodb /var/log/mongodb
sudo chmod 700 /var/lib/mongodb /var/log/mongodb

comment lister les databases , tables existantes dans mongodb

Parfait ! 😎 En MongoDB, on ne parle pas de “tables”, mais de collections (équivalent des tables SQL). Voici comment lister les bases de données et les collections existantes.
1️⃣ Lister les bases de données

Dans le shell MongoDB (mongosh) :

show dbs

💡 Explication :

    Affiche toutes les bases qui contiennent au moins un document

    Si une base est vide → elle n’apparaît pas

Exemple :

admin     0.000GB
local     0.000GB
test_db   0.001GB

2️⃣ Lister les collections d’une base

    Sélectionne la base :

use test_db

    Liste les collections :

show collections

Exemple de résultat :

users
produits

    Ici, users et produits sont des collections (tables en SQL)

3️⃣ Avec Python (pymongo)

from pymongo import MongoClient

client = MongoClient("mongodb://localhost:27017/")
# Lister toutes les bases
print(client.list_database_names())

# Sélectionner une base
db = client["test_db"]

# Lister les collections
print(db.list_collection_names())

Résultat typique :

['admin', 'local', 'test_db']
['users']

💡 Notes importantes

    Une base ou une collection n’existe réellement que si elle contient au moins un document

    show dbs et list_database_names() n’affichent que celles qui ont des documents

    Pour tester : ajoute un document si nécessaire

comment afficher les contenus des collections

Parfait ! 😎 Pour afficher le contenu d’une collection MongoDB, tu peux le faire soit dans le shell MongoDB (mongosh), soit en Python avec pymongo. Voici comment.
1️⃣ Avec le shell MongoDB (mongosh)

    Connecte-toi à la base :

mongosh -u "monuser" -p "monmotdepasse" --authenticationDatabase "test_db"

    Sélectionne la base :

use test_db

    Affiche tous les documents d’une collection :

db.users.find()

    users = nom de la collection

    Renvoie tous les documents sous forme JSON

    Si tu veux un affichage “joli” :

db.users.find().pretty()

Exemple de sortie :

{
  "_id": ObjectId("651b6f8c1f3c4a9a12345678"),
  "nom": "Jean",
  "age": 25
}

2️⃣ Avec Python et pymongo

from pymongo import MongoClient

# Connexion à MongoDB (local, avec utilisateur)
client = MongoClient("mongodb://monuser:monmotdepasse@localhost:27017/test_db")

# Sélection de la base et de la collection
db = client["test_db"]
collection = db["users"]

# Lire tous les documents
for document in collection.find():
    print(document)

💡 Résultat typique :

{'_id': ObjectId('651b6f8c1f3c4a9a12345678'), 'nom': 'Jean', 'age': 25}

3️⃣ Filtrer des documents (optionnel)
Shell MongoDB :

db.users.find({age: 25})

Python :

for doc in collection.find({"age": 25}):
    print(doc)

4️⃣ Limiter le nombre de résultats
Shell :

db.users.find().limit(5)

Python :

for doc in collection.find().limit(5):
    print(doc)

    Chaque document a un _id généré automatiquement

    Tu peux trier, filtrer et projeter les champs que tu veux avec MongoDB, comme en SQL

pour celui ci quand il est dans un fichier start.sh ca ne marche pas source ../venv/bin/activate

Ah oui 😅, c’est un comportement normal sous Linux et Bash :

Le problème vient de la manière dont source fonctionne dans un script.
🔹 Explication

    Quand tu fais source ../venv/bin/activate dans le terminal, ça active l’environnement pour ce terminal.

    Quand tu mets la même commande dans un script start.sh et que tu l’exécutes avec :

./start.sh

ou

sh start.sh

➡️ ça ne modifie PAS ton shell courant.

⚠️ Donc l’environnement virtuel n’est actif que dans le sous-processus qui exécute le script et disparaît dès que le script se termine.
🔹 Solution 1 : sourcer le script

Au lieu de lancer le script normalement, source-le dans ton terminal :

source start.sh

ou

. start.sh

    Le source fait que toutes les commandes du script s’exécutent dans ton shell courant, donc l’activation du venv reste effective après la fin du script.

🔹 Solution 2 : activer le venv directement dans le script pour lancer Python

Si ton objectif est de lancer ton script Python depuis le shell, tu peux le faire en une seule commande dans start.sh :

#!/bin/bash

# Activer le venv
source ../venv/bin/activate

# Lancer ton script Python
python db/mongo.py

Puis dans le terminal :

source start.sh

ou, si tu ne veux pas que le venv reste actif après :

bash start.sh

    Là, le venv est actif pendant l’exécution du script et Python utilisera les bonnes librairies.

    Quand le script se termine, le terminal revient à ton shell normal.

🔹 Solution 3 : utiliser le Python du venv directement

Tu peux contourner complètement source :

#!/bin/bash

../venv/bin/python db/mongo.py

    Là tu n’as pas besoin d’activer le venv

    Python utilisera directement l’environnement virtuel

    C’est plus simple pour les scripts automatisés

💡 Astuce : Pour les scripts de lancement, solution 3 est souvent la meilleure, surtout pour des apps que tu veux exécuter avec un simple ./start.sh.
